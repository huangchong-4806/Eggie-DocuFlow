# Eggie DocuFlow 分阶段稳定整理方案

## 目标

在不改变现有功能、界面布局、文件处理规则和使用习惯的前提下，先建立可靠的检查基线，再把主界面文件中最独立的部分逐步分离。每次只整理一个小范围，检查通过后再继续，避免一次性大改造成按钮、页面状态、后台任务或 PDF 拖拽功能失效。

本方案只覆盖前两个阶段：

1. 建立安全基线和全过程记录。
2. 分离界面样式、后台任务、常用输入控件和 PDF 缩略图控件。

文档处理新旧入口、安全保存方式、各功能页面继续拆分、基础组件升级和安装包优化，均不在本轮实施范围内，后续需要单独出方案、单独确认。

## 已确认原则

- 软件定位为个人使用的办公工具，稳定性优先于追求文件行数或形式上的整齐。
- 不申请 Apple Developer 账号，不做 Apple 公证，不产生公证费用。
- 不改变 PDF 页数上限：页面整理最多 1,000 页，发票解析和文档处理保持 100 页。
- 不升级 openpyxl、pdfplumber、pypdf、Pillow、PySide6 等基础组件。
- 不改变 Excel 合并、Excel 拆分、发票解析、批量改名、文档处理和 PDF 工具箱的业务规则。
- 不改变用户看到的文字、按钮位置、页面顺序、颜色、拖拽方式和操作步骤。
- 不改变 OCR 密钥的保存位置和 `.env` 保护规则，不把任何密钥写入软件文件。
- 只修改本轮整理直接涉及的文件；测试产生的临时文件在检查结束后清理。

## 方案选择

采用“先原样搬出、确认稳定，再减少重复”的方式。

不采用一次性拆分全部页面，因为当前主窗口同时负责导航、后台任务、页面状态和多个工具入口，一次调整范围过大会增加回归风险。也不采用只做少量补丁的方式，因为这不能缓解主界面文件持续膨胀的问题。

## 第一阶段：安全基线

### 1. 固定当前行为

开始修改前记录以下基线：

- 当前版本和工作分支。
- 当前 103 项自动检查的完整结果。
- 七个主要入口可以打开：工作台、Excel 合并、Excel 拆分、PDF 发票解析、文档智能处理、批量改名、PDF 工具箱。
- PDF 页面缩略图可以勾选、拖拽、旋转和预览。
- 图片缩略图可以勾选、拖拽和预览。
- 后台任务运行时有进度提示，任务结束后按钮状态恢复。

现有检查失败时停止整理，先说明失败项目，不在基线不稳定的情况下继续搬动文件。

### 2. 增加结构保护检查

补充少量只针对本轮风险的检查：

- 主程序仍能正确加载被分离的界面样式、后台任务和控件。
- 两类 PDF 缩略图仍使用各自正确的拖拽规则，不会互相串位。
- 页面缩略图旋转后继续复用预览，不重复读取原 PDF。
- 图片缩略图排序后继续显示正确文件名和顺序。
- 后台任务成功、失败时仍能把正确结果传回主窗口。

这些检查只保护现有行为，不增加新功能。

### 3. 建立执行日志

实施时建立一份与本轮整理直接相关的日志，至少记录：

- 操作时间和步骤名称。
- 修改前后的文件位置。
- 每轮自动检查的总数、通过数和失败项目。
- 七个入口的实际打开结果。
- PDF 缩略图的勾选、拖拽、旋转、排序和预览结果。
- Excel 测试文件的来源、输出路径、生成状态和 Microsoft Excel 实际打开结果。
- 软件启动状态、异常提示和临时文件清理结果。

如果某一步失败，日志必须包含失败步骤、涉及文件、实际提示和回退结果，不能只记录“失败”。

## 第二阶段：分离最独立部分

第二阶段严格按以下顺序执行。每完成一项，先运行对应检查；检查失败就撤销本项，不继续下一项。

### 1. 分离界面样式

把现有颜色表和界面样式原样移到独立文件。只改变存放位置，不重写样式，不引入 JSON、YAML 或新的外部组件。

主界面继续通过同一个入口取得完整样式，因此深色界面、主题色和各页面显示应与整理前一致。

### 2. 分离后台任务

把通用后台任务和文档 OCR 后台任务移到独立文件。任务的输入、进度提示、成功结果和失败提示保持不变。

主窗口仍负责决定何时启动任务和如何显示结果，避免本轮同时改动业务流程。

### 3. 分离常用输入控件

把两个常用输入控件原样移到独立文件。外观、数字输入方式和下拉选择方式保持不变。

### 4. 分离 PDF 缩略图控件

先把 PDF 页面缩略图、图片缩略图和对应拖拽区域原样移到独立文件，完成检查后，再单独减少完全相同的部分。

减少重复时只合并以下共同行为：

- 缩略图区域和勾选框的基础布局。
- 勾选和拖拽状态的外观更新。
- 鼠标按下、开始拖拽和拖拽结束的共同处理。
- 拖拽区域接收移动的共同处理。

页面旋转、PDF 页码、图片文件名、双击预览和最终排序仍分别处理，不能为了减少行数强行合并不同规则。

### 5. 主界面保留内容

本轮完成后，主界面文件仍保留：

- 主窗口和侧边导航。
- 所有功能页面的创建与切换。
- 用户选择文件、输出位置和开始处理的操作。
- 各工具的状态显示和完成提示。

本轮不拆分 Excel、发票、文档处理、批量改名等单独页面，防止范围失控。

## 文件安排

计划新增一个只存放界面相关内容的 `ui` 文件夹，避免与现有主界面文件重名：

- `ui/theme.py`：颜色和界面样式。
- `ui/tasks.py`：后台任务。
- `ui/common_widgets.py`：常用输入控件。
- `ui/pdf_widgets.py`：PDF 页面和图片缩略图、拖拽区域。
- `ui/__init__.py`：使上述文件可以被主程序稳定加载。

计划修改：

- `app.py`：删除已经原样移出的内容，并改为从 `ui` 文件夹加载。
- `tests/test_app_navigation.py`：补充本轮结构调整需要的保护检查。

计划新增：

- `docs/optimization_logs/2026-07-15-staged-stability-log.md`：记录全过程和检查结果。

不会修改 Excel 处理、发票处理、PDF 处理、OCR 服务、打包配置、版本号、依赖清单和发布说明。

## 失败处理和回退

- 每个分离步骤单独保存，出现问题时只撤销当前步骤。
- 不使用已失败的整理结果继续下一步。
- 自动检查失败时，记录具体失败项目和提示，然后恢复到上一个全部通过的状态。
- 软件实际启动失败、页面打不开、按钮无响应或拖拽行为异常时，本阶段判定失败。
- Excel 文件出现损坏、修复、恢复或“已修复部分内容”提示时，本阶段判定失败；失败文件删除，重新从原始测试文件生成。
- 不删除用户已有文件，不覆盖现有测试资料，不提交 `.env`、日志中的密钥或测试产生的临时文件。

## 验收方式

### 自动检查

- 当前 103 项检查全部通过。
- 本轮新增的结构保护检查全部通过。
- 每次分离后先运行对应检查，第二阶段结束后再运行全部检查。

### 软件实际检查

- 从当前软件环境实际启动主程序。
- 逐一打开七个主要入口，确认页面内容、按钮和导航状态正常。
- 实际检查 PDF 页面和图片缩略图的勾选、预览、拖拽和排序。
- 运行一个成功任务和一个模拟失败任务，确认进度和失败提示正常。
- 使用现有打包配置在临时目录生成测试版 App，不覆盖 `release` 文件夹中的正式安装包。
- 实际启动临时测试版 App，确认分离后的界面文件已被完整带入安装包；检查完成后删除临时测试包。

### Excel 实际打开检查

- 使用小型测试文件分别生成一份 Excel 合并结果和一组 Excel 拆分结果。
- 用 Microsoft Excel 客户端逐一实际打开生成结果。
- 确认没有“文件损坏”“是否修复”“无法打开或修复”或“Excel 已修复部分内容”等提示。
- 在日志中记录测试文件路径和实际打开结果。

### 文件和清理检查

- 确认只修改方案列出的相关文件。
- 确认 `.env` 没有被提交，软件文件中没有密钥。
- 清理测试生成的临时 Excel、PDF、图片、缓存和日志副本，只保留正式的执行日志。
- 确认工作区没有新增无关文件。

## 完成标准

只有同时满足以下条件，前两个阶段才算完成：

- 所有自动检查通过。
- 源码运行版和临时测试安装包都能够实际启动，七个主要入口能够打开。
- PDF 缩略图相关操作与整理前一致。
- Excel 合并和拆分结果经 Microsoft Excel 实际打开且没有损坏或修复提示。
- 执行日志包含关键步骤、检查结果、生成文件路径和失败排查信息。
- 临时文件已清理，没有修改无关文件。
- 用户看到的功能、界面和操作步骤没有变化。

完成后先汇报改动内容、影响范围、回退方式和测试结果，等待用户确认，再决定是否进入文档处理入口整理或各功能页面继续拆分。
